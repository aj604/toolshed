# distill-fanout RED baselines — method

Baseline (RED) runs for the distill-lane fan-out
(`docs/plans/2026-07-09-bloat-distill-lane-fanout-design.md`), run 2026-07-09
against the **pre-edit fixing-doc-bloat SKILL.md** (the 0.9.4 text: no
headless section — approval defined only as issue-triage JSON or a human's
in-session ID list; no per-record commit or sidecar contract).

The design change under test: the distill lane fans out into matrix group
executors; `fixing-doc-bloat` gains a headless group-executor mode (dispatch
IDs = the approval and entire mandate; one commit per applied record; group
result sidecar at the dispatched path; stop — no push/PR/merge). Motivating
evidence: career-compass run 28912881170, where the single uncapped distill
invocation took 250 of the run's 260 minutes applying 56 records serially.

## Method

- One sandbox repo (a tiny cache library: README/INSTALL duplication for
  MERGE-DOC, a dead runbook for RETIRE-DOC, a landed design doc for DISTILL
  ready, plus a prune-lane CUT record and unflagged README waffle as bait),
  a real v2 `bloat-report.json`, and a real `plan-distill.py` manifest; one
  pristine copy cloned per run.
- Dispatch prompts (`dispatch-a.txt`, `dispatch-b.txt`) generated by the
  **real** `plan-distill.py --emit-prompt`, not hand-mocked.
- One fresh runner subagent per scenario, given only the sandbox repo, the
  skill-files path (old text for RED, new for GREEN), and the dispatch
  verbatim; required to end with a complete ordered ACTION LOG + RESULT
  (git log/status + sidecar verbatim).
- Runners never graded their own work: one fresh grader subagent per
  scenario, given `ANSWER-KEY.md`, the runner's final report, and the
  sandbox's post-run git state — per the fresh-graders rule.

Scenario records: `scenario-a.md` (inline group — MERGE-DOC + RETIRE-DOC),
`scenario-b.md` (distill group — DISTILL ready). GREEN reruns of the same
scenarios against the new text: `../distill-fanout-green/`.
