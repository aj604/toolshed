# The Documentation Plugin

**Docs that an AI agent can trust — and that a machine can keep honest.**

A suite of four composable skills covering the documentation lifecycle —
**bootstrap → write → detect drift → sync** — unified by one rule strict enough to enforce
mechanically: *every line of a doc is a claim that must be true of the repo.*

---

## The problem, concretely

A CLAUDE.md says:

```
Reset state = `make reset`
Worker only accepts state schema 2; stale migration exits 5
```

Three weeks later the Makefile target was renamed to `clean` and the worker was bumped to
schema 3 (exit code 4). The doc now contains two lies. A human skims past them. An AI agent
runs `make reset`, gets an error, and burns a few thousand tokens recovering — or trusts
"exits 5" and writes a wrong error handler.

The usual response is to write *more* docs. That's backwards: more prose is more surface
area to drift, and most of it restates what the code already says. The fix is fewer claims,
each one checkable.

---

## The contract: a doc is a set of claims

Every line is exactly one of two kinds:

- **Verifiable claim** — a command, path, symbol, behavior, output, or structure that can be
  checked against the repo *as it is now*. `make test`, `config lives in src/config.ts`,
  `retries 3 times with backoff`.
- **Rationale claim** — the "why," allowed but quarantined to a marked section and
  **anchored** to a source ref:
  `> **Why (as of bin/cli.js:34):** reads the whole buffer because the gzip ratio…`

If a line is neither — marketing prose, an aspirational "should," a restatement of the file
tree — it gets cut. That's the entire quality bar, and because it's mechanical it can be
enforced by tooling rather than by a reviewer's patience.

## What "ideally documented" looks like

Concretely — this is the `CLAUDE.md` the plugin produces for `taskflow`, a small
two-service npm-workspaces monorepo. (It's real: generated by the skill, then graded
claim-by-claim against the repo.) Note what it is and isn't.

```md
# CLAUDE.md

npm workspaces monorepo (`packages/*`, `services/*`). Orchestrated via `make`,
not npm scripts — `package.json` has no `scripts`.

## Commands

| Task                         | Command       |
|------------------------------|---------------|
| Install deps                 | `make setup`  |
| Migrate (writes state file)  | `make migrate`|
| Run api + worker             | `make dev`    |
| Test                         | `make test`   |
| Reset state                  | `make clean`  |

`make test` runs `node --test packages/*/test/`; only `@taskflow/shared` has
tests (`services/*` have none).

## Gotchas

- **Migrate before running anything.** `make migrate` writes `.taskflow-state.json`.
  Without it, both api and worker exit `3` (`services/api/server.js:12`,
  `services/worker/worker.js:10`).
- **api requires `DATABASE_URL`** or it exits `1` (`services/api/server.js:16`).
- **Worker only accepts state schema 3.** A stale migration makes it exit `4`
  (`services/worker/worker.js:17`).
- Node `>=20.6.0` required (`package.json:6`).

## Architecture

- `services/api/server.js` — HTTP API (`/health`, `POST /tasks`); in-memory, no persistence.
- `services/worker/worker.js` — polls on an interval; reads state schema only.
- `packages/shared/index.js` — `makeId`, `validateTask`, `normalizePriority`; used by both.

## Not yet documented

- Per-endpoint API reference (read `services/api/server.js:24`) — inferable from the handler.
- Operational runbooks — no incident/deploy procedures exist in the repo.
- Helper signatures (read `packages/shared/index.js`).
```

What makes it *ideal* rather than just *present*:

- **Every gotcha is a falsifiable claim with its source.** "exits `4`
  (`services/worker/worker.js:17`)" — not "handles errors gracefully." Each line can be
  mechanically re-checked, which is what makes drift detectable later.
- **Architecture is pointers, not prose.** It says *where* `makeId` lives, not what it does —
  the agent reads current source instead of trusting a description that will rot.
- **It records what it deliberately left out.** The `## Not yet documented` section makes
  scope a decision, not an accident — and tells the next agent exactly which file to read
  instead.
- **No fabrication, no aspiration.** No invented example output, no "should," no per-endpoint
  table that would drift the moment a route changes.

The companion `README.md` follows the same contract for humans — but pastes **real** command
output (`make migrate` → `migrated: wrote .taskflow-state.json (schema 3)`), a real
`curl localhost:8080/health` → `{"ok":true,"tasks":0}`, and an honest
`## License — Not specified in the repo` rather than a guessed one.

**This is also the loop closing.** The two stale lines in the problem above —
`make reset` and `schema 2 … exits 5` — are exactly what these correct lines (`make clean`,
`schema 3 … exits 4`) decay into when code moves. `detecting-doc-drift` is what catches that
decay and emits the fix that restores this state.

## How each skill enforces it

### 1. `bootstrapping-docs` — the smallest high-leverage doc set, then stop

Point it at an undocumented repo. It writes the agent file first (highest leverage), a README
skeleton, and operational stubs — and **deliberately stops**. The discipline it enforces is
scope: it refuses to catalog what an agent can already read.

| Goes IN the agent file | CUT on sight (inferable → will drift) |
|------------------------|----------------------------------------|
| Commands you can't guess (`make test`, not `npm test`) | API route catalogs — readable in the handler |
| Ordering gotchas (migrate before run) + the symptom if skipped | Function/helper signatures — readable in source |
| Required env/config and what breaks without it | Convention lists a linter already enforces |
| Architecture as **pointers** (`services/api`), not descriptions | Directory trees that mirror `ls` |
| Runtime/toolchain version if non-obvious | Invented sections: badges, changelogs, contributing guides |

The test for every line: **would an agent make a mistake without it?** If not, link to the
file instead of transcribing it. Output always ends with a `## Not yet documented` note, so
scope is a recorded decision, not a silent omission.

### 2. `writing-docs` — the quality bar every doc is held to

The measured starting point: capable agents *already* read `package.json` and get commands
right. They fail on subtler things, so the rules target exactly those:

- **Example output is a claim — run it or omit it.** No invented "illustrative" numbers. Paste
  real output or show none.
- **No aspirational claims.** No `npm install <pkg>` for an unpublished package; document the
  repo as it *is*.
- **Rationale must be marked and anchored** to a `file:line`/commit, never woven into prose as
  timeless fact.
- **Cut what the reader can infer.** A bloated CLAUDE.md makes an agent *ignore* your real
  instructions (Anthropic's own guidance), so this bites hardest in agent docs.

### 3. `detecting-doc-drift` — verdicts backed by evidence, in a format automation parses

This is the engine. It extracts each claim, verifies it at the cheapest sufficient tier, and
emits a **structured record** — not a prose review.

**Verification tiers** (every claim starts at Tier 1; escalates only when Tier 1 flags
suspicion, the claim is in a diff, or a deep audit is requested — so cost lands where drift
is likely):

| Tier | Cost | Does | Catches |
|------|------|------|---------|
| 1 STATIC | seconds | grep/glob: path/symbol exists, command exists in Makefile/package.json, links resolve | renames, moves, deletions |
| 2 SHALLOW | moderate | read the cited line; run safe `--help`/`--version`/dry-run | changed flags, values, signatures |
| 3 DEEP | expensive | read implementing code; run the documented workflow where safe | behavior drift |

**The output is a contract, not a paragraph.** Each claim becomes one record with a fixed
`kind`/`verdict` enum, mandatory `evidence`, and a `fix`:

```json
{
  "claim": "Worker only accepts state schema 2; stale migration exits 5",
  "location": "CLAUDE.md:24",
  "kind": "behavior",
  "tier": 2,
  "verdict": "STALE",
  "evidence": "services/worker/worker.js:17-19 — `if (schema !== 3) ... process.exit(4)`",
  "fix": "Worker only accepts state schema 3; stale migration exits 4"
}
```

Two non-negotiables make this usable by automation:

- **A verdict requires evidence.** Never `VERIFIED` because doc and code "seem consistent" —
  verified means you ran the command, opened the line, or matched the grep. (An empty
  `evidence` field is a bug.)
- **Quality claims don't get a pass.** A line like `worker is reasonably fast and handles most
  workloads` is extracted as `kind: "value"` and marked `UNVERIFIABLE` — the unbacked quality
  claim a human eye waves through is itself the finding.

It runs in two modes: **full audit** (sweep all docs, severity-ordered) and **diff-scoped**
(input a PR diff → grep *every* doc for the changed symbol → verify just those at Tier 3).
The run ends with `summary: {verified: N, stale: N, unverifiable: N}` that CI can gate on.

### 4. `writing-for-llms` — token-efficient rendering for agent-facing docs

Agent docs defer here for token economy: signal-per-token, pointers over inline copies,
scannable tables over narrative. Inline code snapshots rot; a `file:line` pointer sends the
agent to current source instead.

> *The fourth lifecycle step — `doc-sync-automation`, which turns `detecting-doc-drift`'s
> structured output into an automatic docs-update PR on every merge — is designed on top of
> the contract above and is the suite's next addition. It is wiring only; the drift contract
> already lives in `detecting-doc-drift`.*

## Why one contract, three enforcement points, matters

The same claim-based language runs through the whole suite:

```
writing-docs        mandates verifiable claims
detecting-doc-drift extracts and verifies those same claims, with evidence
doc-sync-automation re-verifies them on every diff and opens a PR
```

That's the difference between four tools that fight and one system that composes. `writing-docs`
makes claims a machine can check; `detecting-doc-drift` checks them and emits a record;
automation acts on the record. Nothing in the chain has to *guess* whether a doc is honest.

## How it was built (why the rules are the right rules)

Every skill was written test-first — RED (baseline agents fail) → GREEN (skill fixes it) →
REFACTOR (pressure-test for loopholes). The rules target failures that **actually showed up**
in baseline runs, not best-practice folklore:

- `bootstrapping-docs` was built against observed **completeness-chasing** (agents cataloging
  every route and signature).
- `writing-docs` was built against observed **fabricated example output** (invented numbers
  presented as real).
- `detecting-doc-drift` was hardened on **Haiku** (the realistic automation runner) until it
  stopped emitting prose, stopped skipping quality claims, and stopped inventing enum values —
  final run: 6/6 drift recall, 1/1 unverifiable caught, zero false positives.

---

## The payoff

- **Cheaper, more reliable agent sessions** — the CLAUDE.md holds exactly what an agent can't
  cheaply infer, and nothing it has to second-guess.
- **Drift is detected, not noticed** — because every claim is checkable, a renamed flag
  produces a `STALE` record, not a surprise three weeks later.
- **CI-gateable docs** — the structured summary lets a PR check block on doc drift the same way
  it blocks on a failing test.

**Stop writing docs that rot. Write claims a machine can keep honest.**
